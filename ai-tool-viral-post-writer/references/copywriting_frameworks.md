# Copywriting Frameworks for Chinese X/Twitter AI Tool Posts

## Selection rule

Pick one primary framework. Mix only adjacent modules. Never force an analogy from the reference library if the project's own strongest hook is clearer.

## 1. Free/resource information-gap framework

Best for: directories, free APIs, model lists, resource collections, tool roundups.

Spine:
1. Strong hook: “别再一学 AI 就先买订阅了 / 有人把 X 个免费资源整理好了”.
2. Pain: beginner does not know where to find free models/APIs/local tools.
3. Definition: “这不是工具本体，而是一张资源地图/低成本起步清单”.
4. What you get: translate categories into scenarios.
5. What it replaces: tabs, blogs, trial traps, subscription confusion, research time.
6. How to use: open, choose category, test, bookmark.
7. Boundary: free tiers may change, directory not tool, verify before production.
8. Viewpoint: low-cost workflow first, paid upgrade later.

Do not overuse website-directory analogies. Use only if the project truly functions as a curated navigation hub.

## 2. Open-source replacement framework

Best for: local/offline/open-source alternatives to paid SaaS.

Spine:
1. Hook: “某某付费工具要哭了 / 本地开源替代来了”.
2. Pain: cloud fee, API bill, privacy, limited trials, vendor lock-in.
3. Project definition: local/open-source tool for the same job.
4. Capability list: 3-6 tasks it can do.
5. Cost contrast: paid service vs $0/open-source, only if verified.
6. Setup/use path: install/run/import.
7. Boundary: hardware, model quality, setup complexity, license.
8. Opinion: good for learning/prototype/local workflow, not necessarily enterprise replacement.

## 3. Capability breakthrough framework

Best for: new model/tool feature that changes how a task is done.

Spine:
1. Hook: “这个功能把原来很麻烦的流程打穿了”.
2. Before: old workflow was manual/fragmented/slow.
3. Now: new capability in plain words.
4. 3-5 concrete examples.
5. Why it matters: fewer prompts, fewer steps, better context, new workflow.
6. How to try.
7. Boundary: supported platforms, reliability, availability.
8. Viewpoint: from prompting every step to reusable workflow/capability.

## 4. Warning/risk framework

Best for: hidden cost, security, privacy, hardware, policy, or deceptive free-tier issues.

Spine:
1. Hook: “某工具可能正在偷偷消耗你的 X”.
2. Mechanism: explain the technical reason simply.
3. Evidence: cite observable logs, files, settings, docs, issues.
4. Consequence: cost/lifetime/privacy/security impact.
5. What to check.
6. Mitigation.
7. Boundary: avoid panic; say tested condition and uncertainty.

## 5. Agent/workflow framework

Best for: MCP, Codex/Claude/Cursor integrations, browser agents, automation agents.

Spine:
1. Hook: “不是选哪个 AI，而是让几个 AI 各干擅长的事”.
2. Pain: planning/execution/tool access fragmented.
3. Mechanism: agent/plugin/protocol connects tools/environment.
4. Division of labor: model plans, agent executes, tools verify.
5. Use cases: coding, publishing, report, file ops, research.
6. Boundary: permissions, security, rate limits, human review.
7. Viewpoint: AI workflow is moving from single chat to composed systems.

## 6. Build-in-public project framework

Best for: user-built demos, dashboards, prototypes, Spaces, Streamlit, open-source apps.

Spine:
1. Hook: “我做了一个 X”.
2. Problem context.
3. What the product does.
4. Current feature list.
5. Tech stack and data/source credits.
6. Fallback/limitations/disclaimer.
7. Invite feedback/contribution.

## 7. Personal stack/list framework

Best for: “我的 Mac 必备软件 / AI 工具栈 / 本周收藏”.

Spine:
1. Context: who this stack is for.
2. Categories.
3. 1-line why each tool stays.
4. What changed recently.
5. Caveat: not universal, based on current workflow.
6. Ask readers for their stack.
